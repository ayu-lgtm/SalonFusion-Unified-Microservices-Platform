CORS — Central Config CORS is configured ONCE in the gateway. Individual microservices have ZERO @CrossOrigin annotations. Gateway adds DedupeResponseHeader filter to prevent duplicate CORS headers that break browser preflight. 
spring:
  cloud:
    gateway:
      globalcors:
        cors-configurations:
          '[/**]':
            allowedOrigins:
              - "http://localhost:3000"

Feign Client Dependency Map

RestTemplate vs OpenFeign — Why Both?

AspectRestTemplateOpenFeign
StyleImperative — manual HTTP codeDeclarative — just interface
Content-TypeFull control (form-urlencoded ✅)Defaults to JSON only
Service DiscoveryManual URL constructionAuto via Eureka lb://
Used forKeycloak Admin REST APIAll inter-service calls
Why chosenKeycloak needs form-urlencoded POSTClean, readable, no boilerplate
 
// OpenFeign — just an interface, Spring generates implementation
@FeignClient(name = "USER", url = "${USER_SERVICE_URL:}")
public interface UserFeignClient {
    @GetMapping("/api/users/profile")
    ResponseEntity<UserDTO> getUserFromToken(
        @RequestHeader("Authorization") String jwt);

    @GetMapping("/api/users/{userId}")
    ResponseEntity<UserDTO> getUserById(@PathVariable Long userId);
}
// Usage: userFeignClient.getUserFromToken(jwt).getBody();
// No URL hardcoding. lb://USER → Eureka finds it automatically.

RabbitMQ — Async Communication

Event Flow After Payment Success
Why Async here? If Notification Service is down, Payment should still succeed. With RabbitMQ, Payment fires-and-forgets → messages queue up → delivered when Notification Service recovers. No tight coupling. System stays resilient. 
// Queue Configuration (Payment Service)
@Bean public Queue bookingQueue() { return new Queue("booking-queue"); }
@Bean public Queue notificationQueue() { return new Queue("notification-queue"); }

// Jackson2JsonMessageConverter handles Java Object ↔ JSON automatically
@Bean
public RabbitTemplate rabbitTemplate(ConnectionFactory cf, Jackson2JsonMessageConverter converter) {
    RabbitTemplate rt = new RabbitTemplate(cf);
    rt.setMessageConverter(converter);
    return rt;
}

// Producer — fires-and-forgets
rabbitTemplate.convertAndSend("booking-queue", paymentOrder);
rabbitTemplate.convertAndSend("notification-queue", notificationDTO);

// Consumer — in Booking Service
@RabbitListener(queues = "booking-queue")
public void handleBookingEvent(PaymentOrder paymentOrder) {
    bookingService.bookingSuccess(paymentOrder);  // status → CONFIRMED
}

Booking + Payment Lifecycle

1

User Selects Salon + Services + Time Slot

Frontend collects salonId, serviceIds[], startTime

2

POST /api/bookings?salonId=X&paymentMethod=RAZORPAY

Body: {serviceIds: [1,3], startTime: "2025-06-20T10:00:00"} with JWT header

3

Booking Service Calls 3 Services via Feign

UserFeign → get user details · SalonFeign → get salon hours · ServiceOfferingFeign → get prices + durations

4

Slot Conflict Check Algorithm

For every existing booking: overlap = (newStart < existingEnd) AND (newEnd > existingStart). Also validates against salon open/close hours.

5

Calculate Totals + Save Booking (PENDING)

totalPrice = sum of all service prices. endTime = startTime + total duration in minutes

6

Call Payment Service via Feign → Create Razorpay Link

Returns short_url like https://rzp.io/l/abc123. Frontend redirects user to Razorpay.

7

After Payment → Callback

Razorpay redirects to /payment-success/{orderId}?razorpay_payment_id=xxx

8

PATCH /api/payments/proceed?paymentId=xxx&paymentLinkId=yyy

Payment Service verifies via Razorpay SDK. Status = "captured" → SUCCESS

9

Publish to RabbitMQ (Async)

booking-queue → Booking status PENDING→CONFIRMED. notification-queue → WebSocket push to customer + salon owner

WebSocket Real-Time Notifications

// Server — Notification Service
@Configuration
@EnableWebSocketMessageBroker
public class WebSocketConfig implements WebSocketMessageBrokerConfigurer {
    @Override
    public void configureMessageBroker(MessageBrokerRegistry registry) {
        registry.enableSimpleBroker("/notification");
        registry.setApplicationDestinationPrefixes("/app");
    }
    @Override
    public void registerStompEndpoints(StompEndpointRegistry registry) {
        registry.addEndpoint("/ws").setAllowedOriginPatterns("*").withSockJS();
    }
}

// Push to both customer AND salon owner simultaneously
@Component
public class RealTimeCommunicationService {
    private final SimpMessagingTemplate messagingTemplate;

    public void sendNotification(NotificationDTO notification) {
        messagingTemplate.convertAndSend(
            "/notification/user/" + notification.getUserId(), notification);
        messagingTemplate.convertAndSend(
            "/notification/salon/" + notification.getSalonId(), notification);
    }
}
// Client — React (STOMP.js)
const client = new Client({ brokerURL: 'ws://localhost:5000/ws' });

client.onConnect = () => {
  client.subscribe(`/notification/user/${currentUserId}`, (message) => {
    const notification = JSON.parse(message.body);
    showToast(notification.message);  // Real-time toast popup!
  });
};
client.activate();
No Polling! Browser stays connected via WebSocket. When payment is confirmed → RabbitMQ → Notification Service → STOMP push → React toast appears instantly. Both customer and salon owner notified simultaneously from the same event.

Database Design (DB-per-Service)

Database Per Service Pattern Each service owns its own MySQL database. Services NEVER query another service's DB. Data sharing happens ONLY via REST (Feign) or events (RabbitMQ). This is the core principle of microservices data isolation.

user_db · User Service

users

id · email · full_name · phone · role (CUSTOMER/SALON_OWNER) · created_at

salon_db · Salon Service

salons

id · owner_id · name · city · address · open_time · close_time

category_db · Category Service

categories

id · salon_id · name · description

service_db · Service Offering

service_offerings

id · salon_id · category_id · name · price · duration_minutes

booking_db · Booking Service

bookings

id · customer_id · salon_id · service_ids · start_time · end_time · total_price · status

payment_db · Payment Service

payment_orders

id · user_id · booking_id · amount · payment_method · payment_link_id · status

notif_db · Notification Service

notifications

id · user_id · salon_id · booking_id · type · description · is_read

review_db · Review Service

reviews

id · user_id · salon_id · rating · comment · created_at

Local Setup Guide (Step by Step)

Prerequisites Java 17+ · Maven 3.8+ · Node.js 18+ · Docker Desktop · MySQL 8.x · IntelliJ IDEA · VS Code
1

Start Keycloak via Docker

docker run -p 8080:8080 \
  -e KC_BOOTSTRAP_ADMIN_USERNAME=admin \
  -e KC_BOOTSTRAP_ADMIN_PASSWORD=admin \
  quay.io/keycloak/keycloak:26.1.0 start-dev
2

Configure Keycloak (http://localhost:8080)

a. Create Client → Client ID: salon-booking-client → Enable Client Authentication → copy Client Secret
b. Create Client Roles: CUSTOMER and SALON_OWNER
c. Create admin user → assign realm-admin role → turn OFF "Temporary" password
d. Realm Settings → Tokens → Access Token Lifespan: 30 minutes

3

Update user-service/application.yml

keycloak:
  client-id: salon-booking-client
  client-secret: <paste-from-keycloak>
  admin-username: admin@gmail.com
  admin-password: admin
  base-url: http://localhost:8080
4

Start RabbitMQ via Docker

docker run -d --name rabbitmq \
  -p 5672:5672 -p 15672:15672 \
  rabbitmq:3-management
# Management UI: http://localhost:15672 (guest/guest)
5

Create MySQL Databases

CREATE DATABASE user_db;      CREATE DATABASE salon_db;
CREATE DATABASE category_db;  CREATE DATABASE service_db;
CREATE DATABASE booking_db;   CREATE DATABASE payment_db;
CREATE DATABASE notification_db; CREATE DATABASE review_db;
6

Add Razorpay Keys (payment/application.yml)

razorpay:
  api:
    key: rzp_test_xxxxxxxxxx
    secret: xxxxxxxxxxxxxxxxxx

Get from: https://dashboard.razorpay.com/app/keys (Test mode)

7

Start All Backend Services (separate terminals)

# 1. Eureka FIRST
cd eurekaserver && mvn spring-boot:run

# 2. All microservices
cd user-service && mvn spring-boot:run
cd salon && mvn spring-boot:run
cd category && mvn spring-boot:run
cd service-offering && mvn spring-boot:run
cd booking && mvn spring-boot:run
cd payment && mvn spring-boot:run
cd notifications && mvn spring-boot:run
cd review && mvn spring-boot:run

# 3. Gateway LAST
cd gateway-server && mvn spring-boot:run
8

Start Frontend

cd frontend
npm install
npm start
# Open: http://localhost:3000
